1. Key Assumptions & Boundary Conditions

Execution Model

  • Follow Mode: Position changes (not raw trades). We monitor trader positions at a fixed interval and replicate net position changes, reducing execution frequency and slippage.
  • Execution Delay: ~5.0s from signal detection to order placement.
  • Transaction Cost: 8.0 bps one-way (5 bps taker fee + 3 bps estimated slippage).
  • Max Leverage: 3.0x portfolio-level leverage cap.
  • Unified Risk Control: Yes - portfolio-level risk overrides can halt or reduce individual trader exposures.

Data Assumptions

  • Each row represents a daily snapshot of trader account state (equity, cumulative net deposits, PnL).
  • Daily returns are computed as deposit-adjusted equity changes: r_t = (equity_t - equity_{t-1} - deposit_change_t) / equity_{t-1}
  • For multi-day observation gaps, returns are geometrically decomposed to per-day equivalents.
  • Traders with observation gaps > 14 days are flagged for data quality issues.

Operational Constraints

  • Capital: $100,000 initial deployment (small-scale grayscale trial).
  • Cash Reserve: 10% held as buffer for margin and opportunities.
  • Rebalance: Weekly (7-day intervals).
  • No short-selling of the portfolio itself - we only replicate trader positions.

3. Structured Management - Classification & Tiering

3.1 Two-Dimensional Organization

Traders are organized along two dimensions:

  • Horizontal (Style Clusters): K-Means clustering on behavioral features (volatility, autocorrelation, skewness, kurtosis, win rate, activity rate) to identify distinct trading styles. This ensures diversification across uncorrelated return sources.
  • Vertical (Performance Tiers): Composite score ranking determines tier assignment:
    • Core (30%): Highest-quality traders forming the portfolio backbone. Receive the majority of capital.
    • Satellite (40%): Promising traders with higher risk/reward profiles. Provide return enhancement.
    • Watch (30%): Qualified but not yet allocated. Monitored for promotion.

3.2 Composite Score Formula

Score = 0.25 × rank(Sharpe) + 0.15 × rank(Sortino) + 0.10 × rank(Calmar) + 0.15 × rank(Profit Factor) + 0.15 × rank(Consistency) + 0.10 × rank(Win Rate) + 0.10 × rank(Tail Ratio)

4. Capital Allocation

4.1 Allocation Method

Two-level allocation hierarchy:

  1. Tier Level: Fixed ratio - Core 60% / Satellite 30% / Cash 10%
  2. Within Tier: Score-weighted with inverse-volatility adjustment: w_i = (score_i / vol_i) / Σ(score_j / vol_j)

4.2 Concentration Controls

  • Max single trader weight: 10%
  • Max style cluster weight: 40%
  • Correlation-based deduplication for pairs with ρ > 0.7

6.2 Handling Historical Performance Non-Persistence

Historical performance is not guaranteed to persist. Our mitigations:

  1. Walk-Forward Validation: Screening and allocation are designed on in-sample data and validated out-of-sample.
  2. Per-Trader Stop Loss: Limits downside when a trader's edge deteriorates.
  3. Diversification: Multi-cluster allocation reduces dependence on any single trader or style.
  4. Regular Rebalancing: Weekly review allows de-emphasizing underperformers.
  5. Conservative Sizing: 10% cash reserve provides buffer; small initial deployment limits maximum loss.
  6. Watch List Rotation: Watch-tier traders can be promoted as data accumulates, providing pipeline for portfolio refresh.

7.2 Iteration Protocol

  1. Week 1-2: Shadow mode - run signals without real capital, measure execution accuracy.
  2. Week 3-4: 25% capital deployment, monitor slippage and P&L tracking error.
  3. Month 2: Scale to 50% capital if tracking error < 5% and no circuit breaker triggers.
  4. Month 3+: Full deployment with weekly rebalancing and monthly strategy review.

7.3 Promotion/Demotion Rules

  • Promotion (Watch → Satellite → Core): 30-day rolling Sharpe > threshold, no stop-loss triggers, consistent behavior.
  • Demotion (Core → Satellite → Watch → Removed): 2+ stop-loss triggers, Sharpe degradation below threshold, behavioral drift detected.
  • Emergency Removal: Any trader experiencing > 30% single-day loss or suspected manipulation.

8. Technical Implementation Notes

  • Cython Acceleration: The portfolio simulation loop uses Cython (when available) for O(n_days × n_traders) iteration with risk checks, providing ~5-10x speedup over pure Python.
  • Modular Architecture: Each pipeline stage (data loading, feature engineering, screening, classification, allocation, risk management, backtesting) is encapsulated in separate modules for independent testing and iteration.
  • Reproducibility: All random operations use seed=42.
